Robotic Inversion for Adaptive Control

Explores how fixed camera systems within a room can be leveraged to generate real-time environment maps and guide mobile robots by providing them with navigation goals. The project focuses on enhancing spatial awareness and remote path planning using external visual data sources

  • Capture visual data and construct a 2D/3D map of the environment.
  • Track and localize mobile robots within the mapped space.
  • Send movement goals and commands to robots using ROS.
  • Enable path planning and obstacle avoidance using external visual feedback.
Updates
  • Created a map for the mobile robot to navigate the environment.
  • Able to navigate around the map using a user-interactive menu, allowing the user to select various locations to move.
  • Implemented object detection using the YOLO World pre-trained model.
  • Simulated a CCTV environment by mounting a camera in the corner of the room, connected to a Raspberry Pi, streaming over a TCP network. The IP address was then used to run the object detection algorithm on a separate device.
  • Successfully detected objects and published their coordinates.
  • Published the mobile robot’s real-time coordinates, which can be used for obstacle avoidance.
  • Calibrated homography using 4, 6, and 9 points; reduced error from ~2m to ~0.02m in some areas, depending on object distance and visibility.
  • Tested detection consistency between camera-detected obstacles in the costmap and laser scan readings, observing small offsets between the two.
  • Verified that objects which were once detected but are no longer present in the camera frame may temporarily remain in the costmap as “ghost objects” because Nav2 stores the detected object coordinates in a buffer until the map refreshes, after which they disappear.

This mobile robot is called the TurtleBot Create, which is primarily used for educational and research purposes in universities. It can be programmed using ROS2 (Robot Operating System 2), a widely adopted open-source framework that provides libraries and tools for building robot applications. ROS2 enables developers to handle navigation, perception, and control in a modular way, making it an excellent platform for experimentation. I used this robot for testing to evaluate whether my theoretical approach could be applied effectively in a practical environment.
For simulating a CCTV environment, I used a webcam connected to a Raspberry Pi 4B. The Raspberry Pi acted as the edge device, capturing the video stream and running a Python script that broadcast the feed over an IP address using a TCP stream. This video stream was accessible only within the local network. My main device, the TurtleBot, subscribed to this stream to run the object detection algorithm and publish any detected objects to a ROS2 topic.
In this image, I selected six points from the camera (in pixel coordinates) and their corresponding points on the map. Using these, I created a 3×3 homography matrix to transform points from the camera view to the map view.
In this project, I successfully converted the camera’s pixel coordinates into map coordinates. I used YOLO to detect objects, obtained their positions on the map, published these object coordinates to a topic, and visualized them using Matplotlib. The coordinates achieved an accuracy of approximately 90–95%.
During testing, I observed that the camera sometimes hangs after running for 2–3 seconds, and in some cases, it fails to detect objects properly or crashes after consistently detecting objects 5–6 times or after publishing the coordinates on the topic. In this video, I tested the reliability of the program by evaluating whether it could consistently detect objects, publish coordinates every time, and remain stable without crashing. I also tested scenarios such as detecting objects when only a small part is visible in the camera frame, detecting fast-moving objects, and handling situations where an object is temporarily blocked by another object. The results show that if an object is hidden behind a larger object, the camera temporarily loses detection, but once the object reappears in the frame, it is successfully detected again.
In this image, we can see the camera output alongside the RViz visualizer. In RViz, the map of the room is displayed with two black dots. These two dots represent the positions of the detected objects, which were identified using the camera and published as coordinates to the map frame.
In this image, a small white circle marked with a blue arrow shows the object detected and displayed in real time using the camera, then published to the map frame. Alongside this circle, there are also several smaller circles that I refer to as "ghost objects." These appear when the camera had previously detected an obstacle and marked it as an object, but it is either no longer present in the environment or is continuously moving.
In this video, I was testing the difference between the coordinates of detected objects by comparing the camera-detected obstacles in the costmap with the laser scan. I began with 4 map frame coordinates and 4 camera frame pixels to create the homography file, which gave an error of about 2 meters. Increasing the calibration points to 6 reduced the error to 1 meter, and with 9 points the error dropped significantly to around 0.02 meters in some areas. When the object is close to the camera and partially visible, the error is small, but when it is far away and only half visible, the error can increase up to 2 meters. The error also rises if the mobile robot’s localization is slightly misaligned with the map. In the costmap, I am marked as a red circle for reference.
In this test, I wanted to check whether a detected object is only visible in the costmap or also recognized by the mobile robot for navigation. After trying a few locations, I drove the robot from Point A to Point B using a Nav2 goal to see if it would replan when a person appeared between waypoints. To test this, I stood in the path while both the laser and camera updated the map with obstacles, showing a small offset between their detections. The camera marked me as an object and published my position, so I appeared in the costmap. As the robot approached, Nav2 replanned its path with a safe margin, carefully maneuvering right or left to avoid me, and then successfully continued on to the final destination. I am marked as a red circle for reference.
In this test, I attempted to navigate the mobile robot from Point A to Point B without using LiDAR, relying on the camera-detected objects. I created a script that subscribed to the topic providing the planned path coordinates, which could also be visualized in RViz. To test path replanning, I stood in the robot’s path to see whether Nav2 would generate a new path. It successfully replanned; however, the robot did not follow the updated path correctly due to a malfunction that prevented it from moving in reverse. Additionally, I tested how Nav2 handles “ghost locations” that remain in its buffer. By sending Nav2 an empty location string, I observed that it took around 10–12 seconds for the ghost locations to clear. Once cleared, Nav2 generated a new path and the robot successfully reached the target location.
In this test, I repeated the experiment to verify whether the previous result was just coincidence or consistently reproducible. Once again, I was detected and marked as a moving object, with my position continuously updated in both the global map and the costmap. This time, I also experimented with a different approach to clearing the costmap by storing the detected object’s coordinates and checking whether they were still visible in the camera frame. If the object was no longer in the frame, the coordinates were cleared. This method worked in some cases, and the robot successfully avoided me twice by replanning and generating a different path.
This flow diagram shows the complete pipeline of the project: Camera (CCTV) → Raspberry Pi (TCP stream) → YOLO detection → Publish obstacles → Nav2 consumes obstacles → TurtleBot3 replans path → Robot navigates safely. This flow diagram explains how the Robotic Inversion system works from start to finish. The VF090 camera streams video through a Raspberry Pi 4B over TCP, which is processed by a YOLO detection node to identify obstacles. Detected objects are converted into map coordinates and published on the /camera/obstacles topic. Nav2 and RViz2 subscribes to this topic, updating the costmap in real time. The TurtleBot3 then navigates to its goal using predefined or user-selected coordinates. If an obstacle is detected in its planned path, Nav2 replans a new route around it; otherwise, the robot continues to follow the original path until the destination is reached, providing feedback on success.
